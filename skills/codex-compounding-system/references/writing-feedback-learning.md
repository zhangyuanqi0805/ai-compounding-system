# Writing Feedback Learning

This is a template for a local writing-feedback file. Keep it local. Do not paste sensitive transcripts into this file, and do not publish real drafts or private feedback without permission.

Read this before writing any article, first draft, public draft, cloud-document expansion draft, or internal knowledge draft.

## Default Writing Checks

- Start with the reader's real problem, not with an abstract framework.
- Make the core claim obvious in the first one or two paragraphs.
- Use a title that a reader outside the project can understand.
- If a method or framework is mentioned, explain what it is, why it is credible, and why this workflow adapts or extends it.
- Do not write as a daily log. Extract the reusable line, then use the day's case as evidence.
- Mark draft status clearly: `internal draft`, `needs human polish`, or `material draft`.
- Keep a bottom section for evidence and material references when the draft may be expanded later.
- Ask whether the user wants to continue from first draft to polished draft or public draft.
- Ask separately whether the user wants to consider asset-library intake. A writing approval is not an asset-library approval.
- If a file is placed into a stable asset library, use a filename with a source or approval date.

## Suggested Local Table

| Feedback | Better Next Draft |
| --- | --- |
| The opening is too soft. | Put the core judgment first, then explain mechanism and evidence. |
| The title is too internal. | Rewrite as a reader-facing problem and judgment. |
| The draft lacks backing. | Add source, credibility, usage scope, or recognized framework. |
| The draft is not ready for publication. | Mark it as internal or needs human polish. |
| The evidence will be needed later. | Add an evidence and material index at the bottom. |
